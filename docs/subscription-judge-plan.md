# Running the Judge on the Claude Subscription (instead of the API)

**Status:** BUILT & LIVE-TESTED (2026-06-29). Single-judge path, end-to-end.
**Date:** 2026-06-28 (decided) / 2026-06-29 (built + verified on real runs)

## What shipped (2026-06-29)

Data-transfer design (the part that makes it real): the answer text NEVER flows
through the Workflow's args or return value — a real run's dump is multiple MB
(the Fort run is 7.2 MB), which won't fit. Instead `dump` writes one on-disk file;
the Workflow's args carry only `{repo, in_path, start, limit, batch}`; each subagent
fetches its answers off disk via the `batch` subcommand and returns only its small
verdicts. This is why it works on 500+ answer runs.

- `scripts/judge_via_workflow.py` — `dump` (DB pull + exact cache-key computation,
  fact sheet from `audit_runs.fact_sheet` by default, skips already-cached answers);
  `header`/`batch` (stdlib-only so any python can run them — subagents read the
  on-disk file); `inject` (aligns verdicts to keys BY INDEX via `--offset`, so a
  subagent never handles the cache key). Refuses cascade/verify (different keyspace).
- `scripts/prejudge_workflow.js` — the Workflow: one subscription subagent per
  BATCH of answers (default 8), each reading its batch off disk and returning a
  `verdicts` array (record_judgment shape + item index; schema hardcoded, verified
  byte-identical to Python's `_batch_schema()`).
- `.claude/skills/prejudge/SKILL.md` — `/prejudge <run_id>` chains dump → Workflow
  → inject.
- `src/cli.py` — `runs` lists ALL recent runs (no client arg) to find a run_id;
  `judge <run_id>` defaults to the run-row fact sheet (so it hits the prejudge cache
  and assesses accuracy without `--fact-sheet`, matching the UI).
- `src/api/runner.py` + `app.py` — `rejudge_run()` and `POST /audits/{id}/judge`:
  re-judge a stored run through the (now-warm) cache for $0, invalidating the report
  cache. Works in-memory or from storage.
- `web/` — a "Judge" / "Re-judge" button on the report with a "pre-judge on the
  subscription first" hint.

## Live verification (2026-06-29)

Run on three real audits (Anoria/Fort/Oura), each with a source-verified fact sheet
generated by research agents and stored on the run row (`docs/fact-sheet-*.md`):

- **Key parity**: the keys `dump` emits are byte-identical to what a live `Judge()`
  computes (single-judge, stored fact sheet), so the UI/CLI judge hits them.
- **Full loop (Anoria, 10)**: prejudge → inject → `cli judge` and the UI endpoint
  both reported "reused N cached judgments" — $0 API.
- **Scale (Fort, 7.2 MB)**: subscription subagents judged off the on-disk file with
  no answer text in args/return; accuracy flags fire (e.g. competitor-confusion).
- **Accuracy (Oura)**: stale (Ring 4 vs Ring 5) and wrong-pricing ($349 vs $399)
  high-severity flags caught against the new fact sheet.
- **Full prejudge** of Fort (535) and Oura (180) run on the subscription.

Scope is the single judge only (the default config). NOTE: these verdicts are for
dev/testing iteration — they come from a different model (Opus, not temp-0 Sonnet)
and must never feed calibration/gold labeling.

## Token optimization: batching (2026-06-30)

One-subagent-per-answer was too token-hungry. The cost had two multiplicative parts,
both scaling with the answer count N:

1. **Per-subagent scaffolding** — every subagent reloads the harness system prompt +
   tool schemas and does a Bash round-trip. Paid once per *subagent*.
2. **Repeated rubric** — the brand rules + accuracy block + fact sheet are IDENTICAL
   for every answer in a run, yet the old flow re-sent them inside each answer's
   prompt. Paid once per *answer*.

Fix — **batch judging**: one subagent now judges up to `batch` answers (default 8) in
a single call, and the shared rubric is sent ONCE per batch. This attacks both costs
(≈`batch`× fewer subagents; one rubric copy per batch instead of per answer). The
prompt splits into a per-answer HEAD (question + answer) and the shared RUBRIC tail:
`judge.py` derives that split by slicing the untouched `_BASE_INSTRUCTIONS` literal,
so `_single_fingerprint` and the live API judge stay **byte-for-byte identical** —
verified (fingerprint `1e0056…` unchanged). `dump` stores the rubric once as
`preamble` + per-item `{key, body}`; the new `batch` subcommand stitches
`preamble` + N heads into one prompt; subagents return a `verdicts` array tagged with
each answer's item index, which the Workflow scatters back into the positional `raws`
array — so `inject` and the cache keyspace are unchanged. Measured ~63% fewer content
tokens on a 3-answer batch (before the subagent-count savings), and it grows with
batch size and rubric/fact-sheet size. Verified end-to-end: batch prompt renders the
rubric once, inject lands all keys, scatter is robust to out-of-order/dropped/failed
verdicts (they stay un-cached and are re-judged later).

### Adaptive batch size + rate-limit pacing (2026-06-30)

Fixed batch size is a poor fit when answer lengths vary (a batch of 8 long answers can
be huge while 8 short ones waste the amortization). So batching is now **token-balanced**:

- `judge_via_workflow.py plan` greedily packs items `[start, limit)` into batches until
  a batch's per-answer token estimate would exceed `--budget` (default 10000; the shared
  rubric is fixed overhead so the budget bounds only the answer text) or it hits
  `--max-items` (default 16, so a run of tiny answers can't build one batch whose verdict
  *output* explodes). Every batch takes ≥1 item, so a single over-budget answer gets its
  own batch. It's stdlib-only (a ~4-chars/token estimate — approximate is fine for
  balancing). The skill runs `dump → plan → Workflow(batches)`; the plan (a small
  `[{start,count}]` array) rides in args, the answer text still never does.
- The Workflow runs the batches in **waves of `concurrency` (default 4)** so at most that
  many subscription subagents are ever in flight — this paces requests below both the
  harness cap and the subscription's rate limit. Combined with token-sized batches, it
  bounds tokens/min (batch budget) and requests/min (wave width) together. The existing
  MAX_TRIES=3 retry still catches any straggler that gets throttled; lower `concurrency`
  if throttling persists. Verified: plan packs to budget/max-items (and forces 1-per-batch
  under a tiny budget), and the wave runner scatters correctly across out-of-order,
  dropped, and fully-failed batches.

## The problem

Iterating on the pipeline burns real API credit fast (~$40 spent on testing alone).
Most of that spend falls into two buckets that should be treated differently:

1. **Engine queries** (`openai_engine`, `gemini_engine`, `perplexity_engine`, and the
   Anthropic *measurement* engines) — these MUST hit the real APIs. Asking "does
   Claude/GPT/Gemini recommend my client?" means querying the actual model. This is
   the measurement itself; it cannot move off the API.
2. **The evaluation layer** — `judge.py`, plus `content_judge.py`, `offsite/agent.py`,
   and teaser/audit generation. These are *our* calls using Claude as a tool, not
   measuring it. This layer CAN run on the Max subscription instead of the API.

## The decision

**Scope:** Just the judge (`src/pipeline/judge.py`) first. Biggest, cleanest single win.

**Calibration:** Testing only. These subscription-produced verdicts are for dev/testing
iteration. The held-constant judge model does NOT matter here, so we maximize savings.
**Do not** let these verdicts feed calibration / gold-labeling work — that path keeps
using the held-constant API judge (`claude-sonnet-4-5`, `temperature=0`).

## Why the judge is the ideal candidate

The judge is already architected as a **separate offline pass over stored answers** with
a **persistent, content-addressed cache** (`data/judge_cache.sqlite`). The cache key is a
deterministic hash of `(model, prompt_fingerprint, client, competitors, fact_sheet,
prompt, answer)` — see `src/pipeline/judge_cache.py:51`.

The pipeline already checks the cache first and only calls the API on a miss
(`src/pipeline/judge.py:429`). So:

> If we pre-fill `judge_cache.sqlite` with verdicts produced on the **Max subscription**
> (via a Claude Code workflow), then a normal `python -m src.cli judge <run_id>` becomes
> **100% cache hits → $0 API**. No pipeline change required.

## The plan (the "workflow")

1. A script pulls the unique `(prompt, answer)` pairs for a run from the DB, plus the
   client / competitors / fact-sheet.
2. A **Workflow** fans out subagents — one per *batch* of unique answers — each judging
   with the *exact same* system prompt + `record_judgment` tool schema and accuracy
   rules from `judge.py` (the shared rubric sent once per batch), returning structured
   JSON. These agents run on the subscription, not the API.
3. The script computes the real cache key (reproducing `prompt_fingerprint` and
   `JUDGE_MODEL` exactly) and writes each verdict into `judge_cache.sqlite`.
4. Run `judge` as normal → all cache hits → free.

Likely shape: a `scripts/judge_via_workflow.py` (DB pull + cache injection) plus the
workflow itself, scoped to the judge only.

## Caveats (accepted for the testing-only scope)

- **Not byte-identical to the Sonnet temp-0 judge.** The API judge is held constant at
  `temperature=0` on `claude-sonnet-4-5`. Subscription subagents run a different model
  (Opus) and aren't temp-0-pinnable. Fine for dev iteration (often *higher* quality);
  NOT for calibration/gold, where a held-constant judge is the whole point.
- **Batch/offline, not inline.** Good for the testing loop, not for live customer latency.
- **ToS / scale.** Fine for internal development and testing. For *paid client
  deliverables*, the API is the supported path and is a real cost-of-goods to price in.
  Subscription for iteration, API for the final billable run.

## Alternative considered (not chosen)

A judge backend that shells out to the `claude` CLI (Agent SDK, uses subscription login)
so it works inline. Rejected for now: slower and worse at forced structured output than
the batch-workflow approach.

## Out of scope (for now)

- `content_judge.py` (site-audit content scoring)
- `offsite/agent.py` (offsite authority agent)
- teaser / audit generation

These could be routed the same way later, but they're more integrated. Judge first.
